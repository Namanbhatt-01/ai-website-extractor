The Theory of the "Neutral" Bypass
In many CI/CD systems, check statuses are not strictly binary (Pass/Fail). They often include intermediate or alternate states such as Queued, In_Progress, Cancelled, Skipped, or Neutral.

A Neutral Bypass occurs when the security evaluation engine (like a branch protection rule or an external bot) is poorly configured to interpret these non-binary states.

How the Logic Flaw works
If a security policy expects a strict Success status, then a Neutral status will correctly result in a blocked merge. However, if the underlying logic is written as a "negative" check (e.g., IF NOT failure, THEN proceed), a Neutral or Skipped status acts as a pass-through.

The Malicious Input: An attacker submits a commit that is designed to fail testing.
The Status Manipulation: By overriding the CI configuration in the pull request (such as adding if: false, modifying test suites with .skip, or exiting with specific codes like 78), the attacker forces the runner to safely exit before the failing code is reached, emitting a Neutral or Skipped status to the API.
The Logic Gap: If the authorization engine does not explicitly treat Neutral as an unapproved state, it groups it with Success, allowing the malicious code to be merged.
Defending Against State-Machine Bypasses
To prevent Context-Confusion and Neutral Bypasses, CI/CD architectures and branch protection layers must enforce strict atomicity and authenticity rules:

Strict Positive Matching: Rule engines must be designed to require explicit Success signals. Any alternative state (Neutral, Skipped, Unknown) must be automatically bucketed as a failure or require manual review.
Configuration Integrity (Locking CI logic): The most common way attackers force a Neutral state is by modifying the .github/workflows directory or the test suite commands within their pull request.
